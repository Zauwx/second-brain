<objective>
Deliver the thinnest end-to-end authentication slice: a user can register an account and log in to receive a JWT access token plus a refresh token. This creates the `app/auth/` domain package, the `users` + `refresh_tokens` tables, and the register/login endpoints — the foundation every later auth feature builds on.

Purpose: Implements AUTH-01 (register) and AUTH-02 (login → access token). Establishes the JWT issuance and password-hashing machinery (PyJWT + pwdlib[argon2]) and the DB-backed refresh-token table that Plan 02 (refresh/logout) and Plan 03 (per-user isolation) depend on.
Output: A registered user can authenticate via Swagger; `app/auth/` package, two new tables, and a Wave 0 `tests/test_auth.py` scaffold proving register + login behavior against real MySQL.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → /auth/register, /auth/login | Untrusted email + password cross here; validated by Pydantic + verified against DB |
| AuthService → MySQL | Password hash + user row persisted; secret-bearing operations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | Password storage (users.hashed_password) | mitigate | Hash with pwdlib PasswordHash.recommended() (Argon2id); store only the hash; UserRead schema omits hashed_password (asserted in test) — Task 3 |
| T-03-02 | Information Disclosure | Timing attack on login password verify | mitigate | pwdlib Argon2id verify is constant-time; on unknown email still return 401 with same "Invalid credentials" detail — Task 3 login() |
| T-03-03 | Elevation of Privilege | Weak passwords | mitigate | Pydantic @field_validator enforces len>=8 + composition (upper/lower/digit/symbol) → 422 at schema layer (D-13) — Task 3 schemas |
| T-03-04 | Information Disclosure | JWT secret leaked into image/code | mitigate | jwt_secret_key read from settings/.env (extra="ignore"); never hard-coded in committed code or Dockerfile — Task 2 |
| T-03-05 | Spoofing | Forged access token | mitigate | Access token signed HS256 with jwt_secret_key; only sub + exp in payload (no mutable claims) — Task 3 create_access_token |
| T-03-06 | Tampering | npm/pip installs (pyjwt, pwdlib, email-validator) | mitigate | T-03-SC: blocking-human legitimacy checkpoint (Task 1) before `uv add`; packages [ASSUMED] per RESEARCH audit |
| T-03-SC | Tampering | Supply chain — auth packages | mitigate | Task 1 blocking-human verify of PyPI pages; not auto-approvable |
| T-03-07 | Repudiation | Duplicate-account / enumeration via register | accept | 409 on duplicate email reveals existence by design (learning project, D-14); rate limiting deferred (D-06 scope) |
</threat_model>
