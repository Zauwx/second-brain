<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API list endpoint | Untrusted `?page=`, `?size=`, `?sort=`, `?filter=` query params cross into the list handler |
| repository → MySQL | Dynamically-built ORDER BY / WHERE LIKE executed against MySQL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Tampering (SQL injection via ORDER BY) | `?sort=` → repository order_by | mitigate | sort token mapped through a fixed whitelist `{created_at, updated_at}` to ORM column objects; unknown tokens raise ValueError→422; the raw string is never interpolated into SQL |
| T-02-12 | Tampering (SQL injection via filter) | `?filter=` → LIKE clause | mitigate | `Note.content.ilike(f"%{filter}%")` binds the user value as a bound parameter; the f-string only shapes the LIKE wildcards around an already-escaped bound value, not raw SQL text |
| T-02-13 | Denial of Service | large page sizes / deep pagination | mitigate | `size` bounded `le=100` (422 over max, D-05); `page ge=1`. Deep-offset cost acceptable at personal scale; note for Phase 7 if datasets grow |
| T-02-14 | Information Disclosure | error on bad sort | mitigate | bad sort returns a clean 422 with FastAPI default detail (D-19), not a 500 with a stack trace exposing internals |
| T-02-15 | Elevation of Privilege | cross-user list access | accept | endpoints intentionally open this phase (no users); per-user scoping is Phase 3 (AUTH-04) — not introduced here |
</threat_model>
