<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → docker | Only api:8000 crosses to the host; mysql is reachable only on the internal Docker network |
| build-time → runtime | Secrets (MYSQL_*, DATABASE_URL) must enter only at runtime via env_file, never baked into images or committed |
| migration runner → MySQL | Alembic connects with credentials sourced from env, executes DDL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | docker-compose mysql service | mitigate | mysql has no `ports:` block — never exposed to host (D-20); only api:8000 published |
| T-02-02 | Information Disclosure | DB credentials | mitigate | DATABASE_URL / MYSQL_* sourced from `.env` via `env_file` and `os.environ` at runtime; no hardcoded creds in alembic.ini, env.py, or compose (D-20) |
| T-02-03 | Tampering | schema integrity | mitigate | schema built exclusively by versioned Alembic migrations; `Base.metadata.create_all()` forbidden (D-15) so schema state is auditable and reproducible |
| T-02-04 | Denial of Service | api startup vs cold DB | mitigate | `depends_on: condition: service_healthy` + mysql healthcheck prevents api connecting before MySQL is ready, avoiding crash-loop |
| T-02-05 | Spoofing | MySQL image supply chain | accept | `mysql:8.4` is the official pinned image; pin version (not `latest`) for reproducibility; low risk for a self-hosted home lab |
| T-02-SC | Tampering | uv/pip installs (sqlalchemy, asyncmy, alembic) | mitigate | all three are well-known PyPI packages named in CLAUDE.md locked stack; versions pinned with bounded ranges; covered by project research, no `[ASSUMED]`/`[SUS]` packages |
</threat_model>
