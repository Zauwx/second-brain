## 1. System Classification

**System Type:** RAG / Knowledge Q&A (retrieval-augmented generation over a personal note corpus, plus embedding-based related-notes retrieval)

**Description:**
This phase delivers the project's headline feature: a user can ask a natural-language question and receive an answer grounded **only** in their own stored notes, with source note-ID citations, and can retrieve semantically similar notes for a given note. Pipeline: embed the note corpus into chunks → on a question, embed the query → cosine-similarity search over the user's chunks (per-user isolation enforced) → pass top-k chunks above a relevance threshold to the cloud LLM → return a cited answer. "Good" means the answer is faithful to the retrieved chunks (no fabrication), cites the right notes, and refuses ("no relevant notes found") when nothing clears the threshold rather than hallucinating. Users are the note owners themselves (single-tenant-per-user); the corpus is small (personal scale, < 50k chunks).

**Critical Failure Modes:**
<!-- The 3-5 behaviors that absolutely cannot go wrong in this system -->
1. **Cross-user leakage** — an answer or related-notes result surfaces content from another user's notes (violates the per-user isolation invariant established in Phase 3). Catastrophic.
2. **Hallucination / ungrounded answer** — the LLM answers from parametric knowledge instead of the retrieved chunks, or invents citations to note IDs that were not retrieved.
3. **No-answer guardrail failure** — when no chunk clears the relevance threshold, the system produces a confident answer anyway instead of "no relevant notes found" (ROADMAP criterion 2).
4. **Stale embeddings** — editing or deleting a note leaves obsolete chunks in `note_chunks`, so retrieval returns content that no longer exists in the source note (ROADMAP criterion 4).
5. **Citation mismatch** — the answer cites note IDs that don't correspond to the chunks actually used to generate it, breaking user trust in provenance.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Personal knowledge management (PKM) / self-hosted "second brain" note systems. Not a regulated enterprise vertical — this is a single-user personal productivity tool, not healthcare, legal, or financial advisory software. No professional-standards body governs "good" answers here; the bar is set by RAG evaluation practice (RAGAS-style faithfulness/context-precision) plus what the note owner personally considers a faithful reflection of their own writing.

**User Population:** A single account holder per tenant, querying a corpus they personally authored (notes, article clippings, saved links). The user is simultaneously the question-asker, the fact-checker, and the ground-truth authority — they wrote the source material and can verify any answer against it in seconds. This is fundamentally different from enterprise/support RAG where the asker usually cannot verify the answer independently.

**Stakes Level:** Low-to-Medium. No safety-critical, financial, medical, or legal decisions ride on this feature — it is honest to say a wrong answer here costs the user re-reading a note or a moment of confusion, not real-world harm. Do not inflate this into a high-stakes claim just because "AI" is involved. That said, stakes are not zero: the entire value proposition of the app (per PROJECT.md's Core Value) is "retrieve/query my own knowledge in natural language" — if the answer is *not* trustworthy, the core feature is worthless even though no one gets hurt. **Trust/provenance is the one place stakes concentrate**: without accurate citations the user cannot tell a real answer from a plausible-sounding fabrication, and repeated fabrication destroys the entire premise of a tool whose only reason to exist is "know what I actually wrote."

**Output Consequence:** The user reads the answer and either (a) trusts it and moves on without checking, or (b) clicks through the cited `note_id`(s) to verify. Downstream "action" is almost always cognitive (recalling/using one's own past thinking), occasionally copy-paste into other work. There is no automated downstream system consuming the RAG answer in this phase — a human always reads it first.

### What Domain Experts Evaluate Against

**Dimension: Groundedness in the user's own notes (not world knowledge)**
Good: Every factual claim in the answer traces to text actually present in a retrieved chunk; the answer reads as a synthesis of "what I wrote," not a generic explanation of the topic.
Bad: The model fills gaps with its parametric/world knowledge (e.g., user asks "what did I write about Docker networking" and gets a generic Docker networking explainer padded around a thin retrieved chunk) — this is the PKM-specific form of hallucination: not fabricating facts about the world, but silently substituting the model's general knowledge for the user's own notes.
Stakes: Critical
Source: RAGAS "faithfulness" metric — the industry-standard measure of whether a generated answer is supported by retrieved evidence vs. fabricated (Confident AI / RAGAS practitioner guides, 2026).

**Dimension: Citation accuracy and click-back trust**
Good: `cited_note_ids` map exactly to the notes the answer content actually draws from; a user clicking through lands on the note that contains the claim.
Bad: Citations reference notes that were retrieved but not actually used (padding for false confidence), or omit a note that clearly contributed content, or cite a note ID that wasn't retrieved at all.
Stakes: Critical
Source: AI-SPEC Section 1 critical failure mode #5 (citation mismatch) — this rubric ingredient operationalizes it in domain terms: "would the user's click-through match their expectation."

**Dimension: Honest refusal over confident guessing**
Good: When nothing in the corpus is relevant, the system says so plainly ("no relevant notes found") rather than manufacturing a plausible-sounding non-answer.
Bad: A confident, well-formatted answer appears even though the user has no note on the topic — the single most trust-destroying failure for a tool whose entire value is "tell me only what I actually know."
Stakes: Critical
Source: ROADMAP criterion 2 (no-answer guardrail) + RAGAS context recall/precision framing — an empty or low-relevance retrieval set should propagate to a refusal, not be masked by fluent generation.

**Dimension: Freshness — answers reflect the current note, not a stale edit**
Good: If a note was edited or deleted, the answer (and its citation) reflects the current content; an edited note's old chunks never surface.
Bad: The user edits a note, asks a related question, and receives an answer quoting text they already deleted or changed — a distinctly PKM failure mode (the user's own recent edit is the ground truth, and the system contradicts it).
Stakes: High
Source: Cross-referenced against production RAG staleness research — "a document was updated, but the index was not" is documented as one of the most common production RAG issues (Atlan, "LLM Knowledge Base Staleness," 2026); directly maps to AI-SPEC Section 1 failure mode #4.

**Dimension: Retrieval recall across the user's own phrasing gap**
Good: A question phrased differently from the note's original wording (paraphrase, synonym, different framing) still surfaces the right note(s) — this is precisely why semantic (embedding) retrieval was chosen over pure keyword search.
Bad: The system returns "no relevant notes found" or irrelevant chunks for a question about a topic the user clearly does have notes on, just because the wording differs; or, for `/notes/{id}/related`, returns near-duplicate chunks of the same note or misses genuinely related notes because embeddings are too literal.
Stakes: Medium
Source: RAGAS "context recall" — the retrieval-side counterpart to faithfulness, measuring whether the retrieved set actually contains what's needed to answer (RAGAS practitioner guides, 2026); domain framing is PKM-specific (the user's fragmentary, informal writing style widens the phrasing gap between query and note text).

### Known Failure Modes in This Domain

1. **World-knowledge leakage ("the model already knew that").** Because the underlying LLM has broad general knowledge, a thin or ambiguous retrieval result invites the model to quietly pad the answer with things it "knows" generally rather than admitting the user's notes don't cover it. This is the PKM-specific manifestation of hallucination — distinct from inventing false facts, it's substituting *correct but ungrounded* knowledge for the user's own record, which is just as trust-breaking because the user can't tell the difference without checking every citation.
2. **Stale-index answers after edit/delete.** Documented broadly as one of the most common production RAG failures — "a document was updated, but the index was not" — and explicitly called out in this AI-SPEC's Section 1 as failure mode #4. In a personal notes tool this is especially jarring: the user *just* edited the note and expects the system to know.
3. **Fragment-as-fact over-trust.** Personal notes are often short, informal, unverified fragments (a quick thought, a half-formed idea, a pasted quote without full context) — not vetted documents. A faithful-to-context answer can still mislead if it treats a throwaway fragment with the same authority as a carefully written note. This is a domain-specific nuance generic "faithfulness" scoring doesn't capture: faithful-to-the-chunk is necessary but not sufficient when chunks vary wildly in the user's own confidence/care level.
4. **Related-notes near-duplication or triviality.** For `RAG-02`, returning the same note split across two chunks, or a set of superficially similar-but-substantively-unrelated notes (embedding proximity without conceptual proximity), defeats the discovery purpose — the user wanted "notes I forgot I'd connected to this," not "sentences that look alike."

### Regulatory / Compliance Context

No formal regulation applies — this is a self-hosted, single-user-per-account personal tool with no healthcare/financial/legal data-handling obligations (HIPAA, GDPR Art. 9 special-category data, PCI, etc. are not in scope by design; PROJECT.md explicitly excludes managed public cloud hosting and multi-user sharing that would raise these concerns).

One data-egress point is worth flagging honestly rather than as a compliance requirement: **embeddings are computed locally via Ollama (`nomic-embed-text`), but answer generation sends the retrieved note chunk text to Anthropic's cloud API (`claude-sonnet-4-5`)** per the Section 2/4 framework decision. This means the content of a user's private notes — whatever they chose to write — leaves the local machine for every `/ai/ask` call, even though the embedding/retrieval step stays fully local. This is a conscious architecture tradeoff already documented in CLAUDE.md ("IA hybride": local for private/simple tasks, cloud for heavy reasoning) — it is not a violation of anything, but it is the one point where "self-hosted, private second brain" and "cloud LLM call" are in tension, and it's worth the user knowing precisely where that boundary sits if they ever store genuinely sensitive personal content in a note.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Note owner (the user themselves) | The sole ground-truth judge for "does this answer faithfully reflect what I actually wrote" — labels the initial reference dataset (their own real questions against their own real notes) and is the calibration target for any LLM-judge automation. No other "domain expert" exists for this vertical; there is no separate practitioner community to consult since the corpus and its correct interpretation belong entirely to one person. |
| LLM judge (calibrated) | Scales groundedness/citation-accuracy scoring across a larger eval set once calibrated against a handful of the note owner's own pass/fail labels (target ≥0.7 correlation per the eval reference material) — necessary because manual labeling of every RAG interaction does not scale even for a single user, but must never substitute for the owner's judgment on the initial seed set. |

## 2. Framework Decision

**Selected Framework:** LlamaIndex (Python) — used for the RAG pipeline orchestration (ingestion, chunking, query engine, response synthesis). Embeddings via Ollama `nomic-embed-text` (local); answer generation via Anthropic `claude-sonnet-4-5` (cloud).

**Version:** LlamaIndex `0.12.x` (`llama-index-core`), with `llama-index-llms-anthropic` and `llama-index-embeddings-ollama` integration packages. Pin exact versions at plan time via `uv add`.

**Rationale:**
Chosen deliberately at the Phase 6 decision gate, **overriding** CLAUDE.md's original "hand-rolled, no LlamaIndex" recommendation. The user opted for LlamaIndex to get production-grade, batteries-included retrieval (mature chunking/node parsing, query-engine + response-synthesis with citation support) rather than re-implementing the full pipeline by hand. This is the framework the AI Framework matrix ranks #1 for "Production RAG / document Q&A" where retrieval quality is the top priority. **Storage decision:** embeddings are persisted as MySQL JSON columns in a `note_chunks` table with cosine similarity — the pre-Phase-1 decision was retained to avoid a new vector service and to keep the DB pinned at `mysql:8.4` (no MySQL 9.x VECTOR bump, no Qdrant container). LlamaIndex therefore runs against a **custom/simple `VectorStore` backed by MySQL JSON** (or an in-memory index hydrated from MySQL per query) rather than a first-class LlamaIndex vector-store integration — the research agent must design this seam.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Hand-rolled RAG (`anthropic` + Ollama SDKs directly) | CLAUDE.md's original pick (max learning of internals). Ruled out **for this phase** by the user in favor of LlamaIndex's retrieval maturity. Remains the documented fallback if LlamaIndex abstraction proves too heavy. |
| LangChain | Broader/heavier than needed; LlamaIndex is retrieval-first and better fit for RAG-primary. |
| Qdrant vector store | Adds a new Docker service + dependency; overkill at personal scale (< 50k chunks) per CLAUDE.md. |
| MySQL 9.x `VECTOR` type | Requires bumping the pinned `mysql:8.4` to 9.x — unplanned infra scope; rejected in favor of JSON columns. |

**Vendor Lock-In Accepted:** Partial — Anthropic for cloud answer generation (swappable via LlamaIndex's LLM abstraction), Ollama-local for embeddings (no cloud lock-in). LlamaIndex itself is model-agnostic, so provider swaps stay cheap.

### Entry Point Pattern

**Storage seam decision:** implement a **custom `BaseRetriever`** subclass that queries MySQL directly and computes cosine similarity in `numpy` — do **not** implement a full `BasePydanticVectorStore` subclass. Rationale: a full `VectorStore` interface requires implementing `add()`, `delete()`, `query()`, `persist()`, `get_nodes()` etc. for a storage backend LlamaIndex doesn't manage — pure overhead since the app already owns `note_chunks` via SQLAlchemy. A `BaseRetriever` only needs one method (`_aretrieve`) and skips building/persisting a `VectorStoreIndex` object entirely — the app queries MySQL fresh on every request ("hydrate per query" from the AI-SPEC framing). **Tradeoff accepted:** no LlamaIndex-managed ANN index (irrelevant at <50k chunks — brute-force numpy cosine is <50ms), no hybrid/sparse search, no persisted `docstore.json` — all fine at personal scale; would need revisiting past ~100k-200k chunks per user.

```python
# app/ai/rag/retriever.py
import numpy as np
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from llama_index.core.retrievers import BaseRetriever
from llama_index.core.schema import QueryBundle, NodeWithScore, TextNode
from app.models.note_chunk import NoteChunk  # SQLAlchemy model: note_id, user_id, chunk_text, embedding(JSON), ordinal


class MySQLChunkRetriever(BaseRetriever):
    """Retrieves note_chunks for ONE user only — the user_id filter happens in the
    SQL WHERE clause, BEFORE any similarity ranking. This is the per-user isolation
    seam (critical failure mode #1)."""

    def __init__(self, session: AsyncSession, user_id: int, embed_model, similarity_top_k: int = 5) -> None:
        self._session = session
        self._user_id = user_id          # NEVER accept this from request body — from JWT claim only
        self._embed_model = embed_model
        self._similarity_top_k = similarity_top_k
        super().__init__()

    def _retrieve(self, query_bundle: QueryBundle) -> list[NodeWithScore]:
        raise NotImplementedError("Sync path unused — this app is async-first, use _aretrieve")

    async def _aretrieve(self, query_bundle: QueryBundle) -> list[NodeWithScore]:
        # 1. Embed the query (Ollama, local, async)
        query_embedding = query_bundle.embedding or await self._embed_model.aget_query_embedding(
            query_bundle.query_str
        )

        # 2. Pre-filter by user_id in SQL — isolation happens BEFORE similarity math, not after
        result = await self._session.execute(
            select(NoteChunk).where(NoteChunk.user_id == self._user_id)
        )
        chunks = result.scalars().all()
        if not chunks:
            return []

        # 3. Cosine similarity in numpy (personal scale: <50k chunks/user, brute-force is fast enough)
        matrix = np.array([c.embedding for c in chunks], dtype=np.float32)
        query_vec = np.array(query_embedding, dtype=np.float32)
        sims = matrix @ query_vec / (np.linalg.norm(matrix, axis=1) * np.linalg.norm(query_vec) + 1e-10)

        top_idx = np.argsort(-sims)[: self._similarity_top_k]
        return [
            NodeWithScore(
                node=TextNode(
                    id_=str(chunks[i].id),
                    text=chunks[i].chunk_text,
                    metadata={"note_id": chunks[i].note_id, "ordinal": chunks[i].ordinal},
                ),
                score=float(sims[i]),
            )
            for i in top_idx
        ]
```

```python
# app/ai/rag/service.py — wiring: retriever -> explicit threshold gate -> structured LLM call
from llama_index.core import Settings

Settings.embed_model = OllamaEmbedding(model_name="nomic-embed-text", base_url=settings.ollama_base_url)
Settings.llm = Anthropic(model="claude-sonnet-4-5", api_key=settings.anthropic_api_key, max_tokens=1024)

RELEVANCE_THRESHOLD = 0.55  # tune against the Section 5 eval dataset, not a guess

async def ask(session: AsyncSession, user_id: int, question: str) -> "RagAnswer":
    retriever = MySQLChunkRetriever(session, user_id, Settings.embed_model, similarity_top_k=5)
    nodes = await retriever.aretrieve(question)  # QueryBundle built internally by BaseRetriever.aretrieve()

    # Guardrail #3 (no-answer) enforced in CODE, before the LLM is ever called — never rely on the
    # model to "decide" to refuse. Saves an API call too.
    if not nodes or max(n.score for n in nodes) < RELEVANCE_THRESHOLD:
        return RagAnswer(answer="No relevant notes found.", cited_note_ids=[], no_relevant_notes=True)

    return await _generate_structured_answer(question, nodes)  # see Section 4b.1
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `Settings` | Global singleton holding the active `llm` and `embed_model` (replaces old `ServiceContext`) | Set once at app startup (FastAPI lifespan/DI provider); every downstream call reads from it unless overridden per-call |
| `BaseRetriever` (`_aretrieve`) | The pluggable retrieval seam — anything returning `list[NodeWithScore]` | Our custom seam over MySQL `note_chunks`; replaces `VectorIndexRetriever` since there's no `VectorStoreIndex` |
| `NodeWithScore` / `TextNode` | LlamaIndex's node model — text + metadata + relevance `score` | Wrap every MySQL chunk row into one so it flows through `RetrieverQueryEngine` / response synthesizers |
| `SentenceSplitter` | Node parser that chunks note text on sentence boundaries with overlap | Ingestion path only (`POST /notes` create/update) — chunk the note body before embedding + inserting into `note_chunks` |
| `as_structured_llm(PydanticModel)` | Wraps an `LLM` so `.complete()`/`.acomplete()` returns a validated Pydantic instance in `.raw` | The `/ai/ask` answer generation call — needed to get `RagAnswer` with `cited_note_ids` back, not free text |

### Common Pitfalls
1. **Treating `VectorStoreIndex.from_documents()` as required.** Most LlamaIndex RAG tutorials build a `VectorStoreIndex` first. This app has no LlamaIndex-managed vector store (Section 2 storage decision) — skip `VectorStoreIndex` entirely and go straight from the custom `BaseRetriever` to `RetrieverQueryEngine(retriever=..., response_synthesizer=...)`. Following tutorial code literally will pull in an in-memory `SimpleVectorStore` that silently duplicates the embeddings you already persist in MySQL.
2. **`node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=...)]` is NOT the no-answer guardrail.** It silently drops low-score nodes from the context sent to the LLM, but if it drops *all* nodes, `RetrieverQueryEngine` still calls the LLM with an empty context block — Claude will often still generate a plausible-sounding non-answer. The threshold check must happen in application code on `retriever.aretrieve()` results, before any LLM call (see Entry Point Pattern above).
3. **Citations from `CitationQueryEngine` are chunk-text citations, not your `note_id`s.** `CitationQueryEngine` auto-numbers citations `[1]`, `[2]` tied to *citation_chunk_size*-sized sub-splits — it doesn't know about your `note_id` FK. For RAG-01's "cite source note IDs" requirement, don't use `CitationQueryEngine`; instead map `NodeWithScore.node.metadata["note_id"]` yourself and have the structured Pydantic output (`cited_note_ids: list[int]`) reference those IDs directly, then validate them against the retrieved set (Section 4b.1).
4. **`user_id` filtering via LlamaIndex metadata filters (`MetadataFilters`) instead of SQL.** Some LlamaIndex vector-store integrations support post-retrieval `MetadataFilters` for tenant isolation — do NOT use that pattern here. Filtering after an unfiltered similarity search means you fetch, embed-compare, and momentarily hold cross-user chunks in memory before discarding them — an unnecessary blast radius for the isolation invariant. Filter `WHERE user_id = :user_id` in the SQL query itself (see retriever code above), matching the pattern established in Phase 3/4.
5. **Async embedding + async LLM don't compose for free with `asyncio.run()`.** `OllamaEmbedding.aget_query_embedding()` and `Anthropic.acomplete()` are both coroutines — inside a FastAPI async endpoint you `await` them directly. Wrapping either in `asyncio.run()` (copied from a notebook example) inside a running event loop raises `RuntimeError: asyncio.run() cannot be called from a running event loop`. See Section 4b.2.

## 4. Implementation Guidance

**Model Configuration:**
- **Answer generation:** `claude-sonnet-4-5` via `llama_index.llms.anthropic.Anthropic`. `max_tokens=1024` (answers are grounded summaries over a handful of note chunks, not long-form generation — bound it, never leave unbounded). `temperature=0.0` (RAG faithfulness wants deterministic, context-hugging answers, not creative variance — this is a knowledge-retrieval feature, not a writing assistant).
- **Embeddings:** `nomic-embed-text` via `llama_index.embeddings.ollama.OllamaEmbedding`, `base_url` pointed at the internal Docker network Ollama service already running from Phase 5. 768-dim vectors — the `note_chunks.embedding` JSON column stores this as a JSON array of floats.
- Both registered once on `Settings` at app startup (FastAPI lifespan or a DI provider function), not re-instantiated per request — the `Anthropic`/`OllamaEmbedding` clients hold HTTP connection pools.

**Core Pattern:**
Retrieve → code-level relevance gate → structured LLM call. See the Entry Point Pattern in Section 3 for the full `MySQLChunkRetriever` + `ask()` code. Two endpoints share the retrieval half:
- `POST /ai/ask`: embed question → `MySQLChunkRetriever.aretrieve()` (user-scoped) → threshold gate → `as_structured_llm(RagAnswer)` call over the retrieved chunk texts → return `RagAnswer`.
- `GET /notes/{id}/related`: no LLM call at all — embed once using the target note's own stored chunk embeddings (average or first-chunk vector), cosine-compare against the same user's other chunks, group scores by `note_id` (max score per note), return top-N distinct note IDs above a (lower, discovery-oriented) similarity floor. Pure retrieval, no generation — keep it cheap and fast.

**Tool Use:**
No agentic tool-calling in this phase — RAG-01/RAG-02 are a single retrieve-then-generate pass, not a multi-step agent. The only "tool" in the LlamaIndex sense is the structured-output mechanism itself: `Anthropic.as_structured_llm(RagAnswer)` internally issues an Anthropic tool-use call under the hood to force the JSON schema — this is the SDK's implementation detail, not something the app configures directly (verify exact mechanism in `llama-index-llms-anthropic` source if debugging schema-adherence issues).

**State Management:**
Stateless per request — no conversation memory, no `ChatMemoryBuffer`. Each `POST /ai/ask` call is a fresh question with fresh retrieval; there is no multi-turn "chat about my notes" in RAG-01's scope. Persisted state lives entirely in MySQL: `note_chunks` (the embedding index itself) and, optionally, an `ai_queries` audit table (question, retrieved note_ids, answer, timestamp) if Section 7 monitoring wants a query log — not required for the roadmap criteria but cheap to add now given the existing async SQLAlchemy session is already in scope.

**Context Window Strategy:**
`claude-sonnet-4-5` has a 200K-token context window — at `similarity_top_k=5` and note chunks sized ~512 tokens (`SentenceSplitter(chunk_size=512, chunk_overlap=50)`), the retrieved context is ~2.5K tokens, nowhere near the limit. The strategy is therefore **retrieval-size discipline, not truncation**: keep `similarity_top_k` small (5–8) and rely on the relevance threshold to exclude noise, rather than stuffing the window with marginal chunks "just in case." See Section 4b.4 for the explicit token-budget guard to add anyway (defense in depth if a user's note corpus grows and `top_k` is raised later).

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

```python
# app/ai/rag/schemas.py
from pydantic import BaseModel, Field


class RagAnswer(BaseModel):
    """The structured contract for POST /ai/ask. Every field is required so the
    caller can never silently receive a half-populated response."""

    answer: str = Field(..., description="The grounded answer, or a refusal message if no_relevant_notes=True")
    cited_note_ids: list[int] = Field(
        default_factory=list,
        description="note_id values the answer draws from. MUST be a subset of the retrieved chunk note_ids "
        "— validated in application code after parsing, not trusted from the model.",
    )
    no_relevant_notes: bool = Field(
        default=False, description="True when retrieval found nothing above the relevance threshold"
    )
```

**Framework integration:** `llama_index.llms.anthropic.Anthropic` supports `as_structured_llm(RagAnswer)`, which wraps the LLM so `.acomplete(prompt)` returns a response whose `.raw` attribute is a validated `RagAnswer` instance (uses Anthropic tool-calling under the hood to force schema-conformant JSON — confirmed via `llama-index-llms-anthropic` docs, see Section 3 Sources). Build the prompt from the retrieved `NodeWithScore` texts + the question:

```python
async def _generate_structured_answer(question: str, nodes: list[NodeWithScore]) -> RagAnswer:
    context_block = "\n\n".join(f"[note_id={n.node.metadata['note_id']}] {n.node.text}" for n in nodes)
    retrieved_note_ids = {n.node.metadata["note_id"] for n in nodes}

    prompt = PromptTemplate(
        "You answer ONLY using the notes below. Cite every note_id you actually used.\n"
        "If the notes don't answer the question, set no_relevant_notes=true.\n\n"
        "Notes:\n{context_block}\n\nQuestion: {question}"
    )

    sllm = Settings.llm.as_structured_llm(RagAnswer)
    raw_answer = await _predict_with_retry(sllm, prompt, context_block=context_block, question=question)

    # Domain-level validation the schema alone can't enforce: citation mismatch is critical failure mode #5
    invalid_ids = set(raw_answer.cited_note_ids) - retrieved_note_ids
    if invalid_ids:
        structlog.get_logger().warning("rag.citation_mismatch", invalid_ids=list(invalid_ids), question=question)
        raw_answer.cited_note_ids = [nid for nid in raw_answer.cited_note_ids if nid in retrieved_note_ids]
    return raw_answer
```

**Retry logic:** wrap the `sllm.acomplete()` call with `tenacity` (already a project dependency from Phase 5) — retry up to **3 attempts**, exponential backoff (`wait_exponential(multiplier=0.5, max=4)`), retrying on Pydantic `ValidationError` (malformed/incomplete JSON from the model) and on Anthropic rate-limit errors. Log each retry attempt at `warning` level with the raw failure reason via `structlog`. If all 3 attempts fail, surface a 502-equivalent app error to the caller rather than returning a fabricated `RagAnswer` — never fall back to a free-text string wearing the Pydantic model's shape. The citation-mismatch filtering above is a **separate, non-retrying** post-validation step — an invalid `note_id` is not a schema failure worth spending another API call on; log it and strip it.

### Async-First Design

LlamaIndex is async-native for this stack: `OllamaEmbedding.aget_query_embedding()`, `BaseRetriever.aretrieve()` (calls your `_aretrieve` override), and `Anthropic.acomplete()`/`.achat()` are all coroutines that compose directly with FastAPI's `async def` endpoints — just `await` each in sequence inside `app/ai/rag/service.py`. **The one common mistake:** LlamaIndex's own notebooks demonstrate patterns like `asyncio.run(main())` because notebooks aren't already inside an event loop. A FastAPI request handler *is* already inside `uvicorn`'s running event loop — calling `asyncio.run()` anywhere in that call path raises `RuntimeError: asyncio.run() cannot be called from a running event loop`. Copy the `await` calls from the docs, never the `asyncio.run()` wrapper around them. **Stream vs. await:** always `await sllm.acomplete()` (not `astream_chat`) for `/ai/ask` — a `RagAnswer` must be fully parsed and Pydantic-validated before any of it can be trusted or returned; streaming partial JSON tokens can't be validated mid-stream. Streaming would only make sense for a future free-text chat UI, which is out of scope for RAG-01/RAG-02.

### Prompt Engineering Discipline

Keep the system framing ("you answer ONLY from the notes below; cite every note_id used; refuse if nothing is relevant") separate from the per-request user content (the retrieved chunk block + the question) — pass the instruction as the fixed prefix of the `PromptTemplate` and interpolate only `{context_block}` and `{question}` per call, so the instruction can't be diluted or overridden by note content that happens to contain prompt-like text (a real risk when notes are user-authored freeform text). No few-shot examples needed for this task — it's a narrow extraction-style task (answer + cite + refuse), not a style-sensitive generation task, and Pydantic schema validation already constrains the output shape more reliably than examples would. `max_tokens=1024` is set explicitly on the `Anthropic` LLM instance (Section 4 Model Configuration) — never leave it unbounded, both for cost control and because an unbounded response makes structured-output parsing failures harder to diagnose (truncated JSON vs. malformed JSON look identical without a hard cap to rule out truncation).

### Context Window Management

RAG-specific strategy (per Section 4 Context Window Strategy): retrieval-size discipline is the primary control (`similarity_top_k=5-8`, chunk size 512 tokens), but add an explicit **token-budget guard as defense in depth** — before building `context_block`, sum estimated tokens across the retrieved nodes (character count / 4 is an adequate approximation at this scale; add `anthropic.count_tokens` via the SDK if precision is needed later) and if the sum exceeds a budget (e.g., 8K tokens — generous headroom under the 200K window while still catching a pathological case like an unusually large `similarity_top_k` set at runtime), drop the lowest-scoring nodes first until under budget, never truncate mid-chunk-text (that would produce a citation whose text doesn't match what's actually in the note). This guard is cheap insurance against a future `top_k` increase or a user with an unusually large corpus silently blowing the budget.

### Cost and Latency Budget

At `claude-sonnet-4-5` pricing (~$3/MTok input, ~$15/MTok output — verify against current Anthropic pricing page at implementation time, rates change): a typical `/ai/ask` call sends ~2.5K tokens of retrieved context + prompt overhead (~3K input tokens) and returns a short structured JSON answer (~200-400 output tokens) → **roughly $0.009-$0.015 per question** ($3×3000/1e6 + $15×300/1e6 ≈ $0.009 + $0.0045). At personal-project usage (single user, tens of questions/day, not thousands), this is negligible — no caching layer is justified for v1. **Do add later if usage grows:** exact-match caching (hash of `user_id + normalized question`) for repeated questions, since personal knowledge-base questions often repeat ("what did I write about Docker networking"). **Sub-task routing:** embeddings already run on local Ollama (`nomic-embed-text`, zero marginal cost) — the only cloud-billed call in the whole pipeline is the final structured-answer generation; there's no cheaper sub-task to route away from `claude-sonnet-4-5` here (no classification/routing/summarization pre-step in this pipeline) — if a future "related notes via LLM re-ranking" feature is added, that would be the candidate for `claude-haiku-4-5` per CLAUDE.md's model reference table. Latency: local `nomic-embed-text` embed (~50-200ms on CPU per Phase 5's documented Ollama CPU latency) + MySQL fetch/cosine (<50ms at this scale) + Anthropic round-trip (~1-3s typical for a ~1K-token structured completion) → expect **~1.5-3.5s end-to-end** for `/ai/ask`, well within a synchronous HTTP request/response (no need for BackgroundTask/polling, unlike the Phase 5 Ollama summarization path which had to account for much slower local CPU inference).

## 5. Evaluation Strategy

> Grounded in Section 1 critical failure modes (#1 cross-user leakage, #2 hallucination, #3 no-answer gate, #4 stale embeddings, #5 citation mismatch) and the Section 1b rubric ingredients (groundedness, citation trust, honest refusal, freshness, retrieval recall). The note owner is the sole ground-truth authority (§1b) — engineers build the harness, the owner labels the seed set.

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Per-user isolation** (§1 FM#1) | PASS: every chunk considered, ranked, and cited for user A's question belongs to user A — verified because the `user_id` filter is in the SQL `WHERE` clause (§3 `MySQLChunkRetriever`), *before* similarity math. FAIL: any chunk, note_id, or answer sentence traceable to user B's corpus surfaces for user A. | **Code** (deterministic): seed each user with distinct notes, assert a cross-user question returns 0 cross-tenant note_ids and an empty/refusal answer. Runs in CI with mocks (criterion 5). | **Critical** |
| **Citation accuracy — subset invariant** (§1 FM#5) | PASS: `cited_note_ids ⊆ {retrieved note_ids}` and a click-through lands on a note that actually contains the claim. FAIL: a cited note_id was never retrieved, or a note that clearly contributed content is omitted. | **Code** for the subset invariant (§4b post-validation strips invalid IDs + logs `rag.citation_mismatch`); **LLM-judge/Human** for the softer "click-through matches expectation" on the offline set. | **Critical** |
| **Honest refusal — no-answer gate** (§1 FM#3) | PASS: when no chunk clears `RELEVANCE_THRESHOLD` (§3), the response is `no_relevant_notes=true` with an empty `cited_note_ids` and a plain refusal — the LLM is never called. FAIL: a confident, well-formatted answer appears for a topic the corpus does not cover. | **Code** (deterministic): feed a query whose top cosine score is below threshold, assert `no_relevant_notes=true` and zero LLM calls. Runs in CI with mocks. | **Critical** |
| **Context faithfulness / groundedness** (§1 FM#2, §1b groundedness) | PASS: every factual claim traces to a retrieved chunk; reads as a synthesis of "what I wrote," not a generic topic explainer. FAIL: the model pads with parametric/world knowledge the notes do not contain (the PKM-specific hallucination). | **RAGAS `faithfulness`** + calibrated **LLM-judge** on the offline harness (real services). Human note-owner labels the calibration seed. | **Critical** |
| **Retrieval quality — recall across phrasing gap** (§1b retrieval recall) | PASS: a paraphrased/synonym-worded question still surfaces the right note(s); the retrieved set contains what's needed to answer. FAIL: "no relevant notes" or off-topic chunks for a topic the user demonstrably has notes on. | **RAGAS `context_recall` + `context_precision`** on the offline harness against owner-labeled expected note_ids. | **High** |
| **Answer relevance** (§1b) | PASS: the answer addresses the actual question asked, not an adjacent topic present in the retrieved chunks. FAIL: answer drifts to what was retrieved rather than what was asked. | **RAGAS `answer_relevancy`** (offline). | **High** |
| **Freshness — no stale chunks** (§1 FM#4, ROADMAP crit. 4) | PASS: after a note is edited/deleted, its old chunks never surface; the answer + citation reflect current content. FAIL: an answer quotes text the user already changed or deleted. | **Code** (deterministic): create → embed → edit/delete note → re-query old content → assert 0 stale chunks returned. Runs in CI with mocks. | **High** |
| **Related-notes quality — distinct & non-trivial** (RAG-02, §1b FM#4) | PASS: `/notes/{id}/related` returns *distinct* notes (grouped by `note_id`, max score per note per §4), semantically related, excluding the source note itself, above the discovery floor. FAIL: two chunks of the same note, the source note echoed back, or embedding-proximate-but-conceptually-unrelated notes. | **Code** for the structural checks (distinct note_ids, source excluded, near-duplicate suppression); **LLM-judge/Human** for "genuinely related" on the offline set. | **Medium** |

> **Fragment-as-fact caveat (§1b FM#3):** faithfulness scoring confirms an answer is faithful *to the chunk* but cannot detect when a chunk is itself a throwaway/unverified note being treated with unearned authority. This is a known limitation flagged for the note-owner's human review, not an automatable metric in v1.

### Eval Tooling

**Primary Tool:** **RAGAS** (RAG metrics: faithfulness, answer_relevancy, context_precision, context_recall) for the offline quality dimensions + **Arize Phoenix** (open-source, self-hostable via Docker — matches the home-lab/Docker ethos over hosted Langfuse) for OpenTelemetry tracing of the retrieve → gate → generate pipeline. Deterministic dimensions (isolation, citation-subset, no-answer gate, freshness) are plain **pytest** code checks — no external tool needed.

> **Test-suite boundary (ROADMAP criterion 5 — load-bearing):** the pytest suite mocks the cloud LLM (`Anthropic`) *and* the embed service (`OllamaEmbedding`) — ZERO real Anthropic/Ollama calls in CI. Therefore the **code-based** dimensions (isolation, citation-subset, no-answer gate, freshness, related-notes structure) run *inside* the mocked pytest suite. The **RAGAS / LLM-judge** dimensions (faithfulness, answer relevance, context recall/precision) require real model output and real embeddings, so they run in a **separate offline eval harness** (`evals/`), invoked manually or on a schedule against live services — never inside the mocked unit/integration `pytest` run.

### Reference Dataset

**Size:** Start at **12–20 examples** (per eval-reference "10–20 high-quality over 200 mediocre"); grow only from real production failures, not hypothetical coverage.

**Composition:** each example = `{question, seeded_notes, expected_note_ids, expected_no_relevant_notes, gold_answer_notes}`. Cover:
- **Grounded happy path** — question with a clear supporting note (faithfulness + citation).
- **Paraphrase / phrasing-gap** — question worded differently from the note (retrieval recall, the reason semantic search was chosen).
- **No-relevant-notes** — question on a topic the corpus genuinely lacks (refusal gate must fire).
- **Cross-user isolation** — user A's question against a corpus where user B holds the only relevant note (must refuse, never leak).
- **Stale-after-edit** — note edited/deleted post-embedding, then queried for the old content (freshness).
- **Related-notes traps** — a note whose corpus contains (a) a near-duplicate second chunk of itself and (b) an embedding-similar-but-unrelated note (distinctness + conceptual relevance).
- **Fragment-as-fact** — a thin/informal note flagged for human judgment on over-trust.

**Labeling:** The **note owner** is the sole ground-truth authority (§1b) — they author the seeded notes, ask their own real questions, and set the pass/fail label. A **calibrated LLM judge** then scales groundedness/citation scoring across the set once it hits **≥ 0.7 correlation** with the owner's labels (eval-reference target); it must never replace the owner's judgment on the seed set. Build the dataset *concurrently* with implementation, not after.

## 6. Guardrails

> Online guardrails are the deterministic §4 mechanisms — catastrophic if wrong, so they run in code on every request, never delegated to the model. Offline flywheel = sampled batch quality signals feeding refinement.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Per-user SQL pre-filter** (§3 `MySQLChunkRetriever`, §1 FM#1) — `user_id` from JWT claim, never request body | Every `/ai/ask` and `/notes/{id}/related` retrieval | **Block by construction**: `WHERE user_id = :user_id` filters *before* similarity ranking, so cross-tenant chunks are never fetched, embedded-compared, or held in memory. Isolation is structural, not post-filtered. |
| **No-answer relevance-threshold gate** (§3 `RELEVANCE_THRESHOLD=0.55`, §1 FM#3) | `max(node.score) < RELEVANCE_THRESHOLD` or empty retrieval | **Block generation**: return `RagAnswer(no_relevant_notes=True, cited_note_ids=[])` in application code *before* the LLM is called — never rely on `SimilarityPostprocessor` (it still calls the LLM with empty context per §3 pitfall #2). |
| **Citation-subset validation** (§4b `_generate_structured_answer`, §1 FM#5) | Parsed `cited_note_ids` contains an id ∉ retrieved note_ids | **Flag + strip**: log `rag.citation_mismatch` (structlog, warning) and drop the invalid ids — a non-retrying post-validation step (not worth another API call). |
| **Structured-output validation + bounded retry** (§4b) | Pydantic `ValidationError` / malformed JSON / Anthropic rate-limit | **Retry ≤3** (tenacity, exp backoff) then **escalate**: surface a 502-equivalent error — never return a free-text string wearing the `RagAnswer` shape. `max_tokens=1024` caps truncation ambiguity. |
| **Token-budget guard** (§4b defense-in-depth) | Estimated context tokens > ~8K (e.g. a raised `top_k`) | **Flag + trim**: drop lowest-scoring nodes until under budget; never truncate mid-chunk (would desync a citation from its note text). |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **RAGAS faithfulness** (groundedness, §1 FM#2) | Weekly batch over the reference set + a sampled slice of real `/ai/ask` traces (weight toward citation-mismatch flags & low-top-score answers) | Below owner-calibrated bar → tighten the "answer ONLY from notes" prompt (§4b) and/or raise `RELEVANCE_THRESHOLD`; re-calibrate judge. |
| **RAGAS answer_relevancy** | Same batch | Persistent drift → inspect prompt / retrieval `top_k`. |
| **RAGAS context_precision / context_recall** (retrieval, §1b recall) | Same batch against owner-labeled `expected_note_ids` | Low recall → revisit chunk size/overlap (`SentenceSplitter`) or embedding choice; low precision → raise threshold / lower `top_k`. |
| **Related-notes distinctness rate** (RAG-02) | Sampled `/notes/{id}/related` calls | Rising near-duplicate/source-echo rate → adjust dedup-by-note_id grouping or discovery floor. |
| **Citation-mismatch rate** | All `rag.citation_mismatch` log events (already emitted online) | Rising trend → prompt regression; treat as an early warning of faithfulness decay. |

## 7. Production Monitoring

**Tracing Tool:** **Arize Phoenix, self-hosted** (open-source, runs as a local/Docker service, OpenTelemetry-based, no external account — chosen over hosted Langfuse to match the project's self-hosted home-lab ethos). Instrument LlamaIndex via `LlamaIndexInstrumentor()` so every `/ai/ask` trace captures the full retrieve → gate → generate span tree (query embedding, per-chunk scores, threshold decision, structured LLM call, citation validation). At personal scale this doubles as the debugging surface; the optional `ai_queries` audit table (§4 State Management) can persist question / retrieved note_ids / answer / timestamp for durable review.

**Key Metrics to Track:**
1. **No-relevant-notes rate** — `% of /ai/ask returning no_relevant_notes=true`. A spike may signal an embedding/ingestion regression (right notes stopped being retrievable); a floor near 0% may signal the threshold is too permissive (hallucination risk).
2. **Citation-mismatch rate** — frequency of `rag.citation_mismatch` warnings; leading indicator of faithfulness decay.
3. **Retrieval top-score distribution** — the `max(node.score)` per query relative to `RELEVANCE_THRESHOLD`; drift downward hints at stale/degraded embeddings.
4. **End-to-end latency** (p50/p95) — target ~1.5–3.5s per §4b budget (Ollama embed + MySQL cosine + Anthropic round-trip).
5. **Cost per ask** — Anthropic input/output tokens per `/ai/ask` (~$0.009–$0.015 expected per §4b); the only cloud-billed call in the pipeline.

**Alert Thresholds:** (personal-scale — surface in logs/dashboard, not a pager)
- Citation-mismatch rate > 5% of asks over a rolling window → investigate prompt regression.
- p95 latency > 6s sustained → check Anthropic API / Ollama CPU health.
- Structured-output retry exhaustion (502s) > 1% of asks → schema-adherence or provider-availability issue.
- Cost per ask > ~2× the $0.015 baseline → runaway `top_k`/context (token-budget guard should have caught it — investigate).

**Smart Sampling Strategy:** Weight human (note-owner) review toward interactions carrying concerning signals rather than uniform sampling: any `rag.citation_mismatch` flag, any answer where `max(node.score)` sat just above threshold (borderline-relevant), any structured-output retry, unusually long/short answers, and any `no_relevant_notes=true` on a topic the owner believes they *do* have notes on (a retrieval-recall miss). Watch for **signal-metric divergence** — e.g., faithfulness scores stay green while the owner keeps clicking through citations to verify — as the early warning that the eval set has a gap.
