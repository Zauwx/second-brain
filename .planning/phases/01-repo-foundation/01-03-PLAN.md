<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build context -> Docker image layers | Files in the context can be baked into shareable/pushable image layers, including secrets |
| host -> container (runtime env) | Secrets must cross only at runtime via env_file, never via build args or COPY |
| local working tree -> PUBLIC GitHub history | The first push makes history permanent and world-readable; a leaked secret is irreversible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Information Disclosure | `.env` baked into Docker image layers | mitigate | `.dockerignore` excludes `.env`; Dockerfile has no `COPY .env` / secret `ENV` (Task 1); secrets passed only via compose `env_file` (Task 2) |
| T-01-07 | Information Disclosure | Real `.env` committed in the first push to a PUBLIC repo (irreversible) | mitigate | `.gitignore` from Plan 01 ignores `.env`; Task 5 verifies `git status` pre-commit and asserts `git log --all -- .env` returns nothing post-push (blocking) |
| T-01-08 | Spoofing / Elevation | Unauthenticated or wrong-account gh push creates the repo under the wrong identity | mitigate | Blocking human-action checkpoint (Task 4) confirms `gh auth status` before any `gh repo create`/push |
| T-01-SC | Tampering | `uv` package installs during image build (supply chain) | accept | Image installs the same vetted Phase-1 deps from `uv.lock` (`uv sync --frozen`); pinned lockfile prevents drift; no new packages introduced here |
</threat_model>
